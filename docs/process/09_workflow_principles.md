# 09 · 工作流原則（通用方法論）

多角色信箱工作流實戰蒸餾的核心原則。專案無關，任何用此 template 的專案都適用。

## 1. 無斷點自動鏈

角色收 handback → **做完 + 推下一站**（inbox-watch 自動喚下一角色）。**禁自造斷點**：不「park／排隊／下個 session 再做／擱著」。

- **只有需用戶裁決時才停鏈**——真決策：願景 fork／授權／喬不攏的優先序。其餘角色間自動鏈到底。
- 要用戶輸入時給**具體待裁問題**，非「要不要繼續」。判斷角色尤其別給用戶「①②③選哪個」停頓菜單。
- 用戶要的是機器持續轉，只在岔路口喊他。

## 2. Measure-first（框內紀律）

- **先量測再開藥**——別在腦內定因果。
- **相關 ≠ 因果**：某指標隨改動變 = 相關；要斷因果得做**因果檢定**（逼/擋那路徑看指標動不動），別把相關喊成 ironclad。
- **單樣本 ≠ 真**：單 seed/單 run 的結論可能是該樣本的幽靈。**多 seed/多樣本**才斷方向。
- **全探針一次抓全** > 反應式逐維補量（避免「量了好幾輪錯維度才找到真的」）。

## 3. 補丁閘優先查（框內診斷預設）

遇「某行為缺失／塌陷／從不 fire」→ **第一件事查是不是補丁閘**：硬寫的 gate／override／continue／絕對門檻 **pre-empt 掉了本該由引擎/邏輯做的決策** → **先於**猜「參數沒調好／設計沒做／本該如此」。

- 找到 → **de-patch**（決策交引擎/正常路徑，讓它 pre-empt 機械閘），**非加補償補丁**（補丁疊補丁）。
- 「行為不見」常是補丁 pre-empt，非真缺。錯把補丁病當 tuning 病 → 白調參數。

## 4. 框外挑框（降 groupthink，最深的洞）

**問題**：判斷層若同一個模型（如都 Opus），分開的 session 也共享 priors → 獨立也推到同一（可能錯）結論。模型多樣若只在**下位機械角色**（產數字/跑測），且弱模型 defer 上位確信框架 → 碰不到判斷層。

**自我質疑的硬邊界**：能驗數據/執行（重跑/重查），**驗不了自己的框架**——同 priors 生的詮釋，自驗還是同結論。**框外審框在 priors 內部做不到**，結構上需異質對手 or 人。

**藥：選擇性召異質 skeptic 挑框（非全審）**。★觸發三對齊才召：
1. 下**強結論且 redirect 大量工作**
2. **相關跳因果**
3. **覺得 ironclad/很確定**（★高信心＝危險信號）+ **難逆**（build/ship/merge）

- 其餘直接過＝省。**放早**（第一次下大框 call 時）prevent 白工；無差別亂審才浪費。
- 分層：便宜先**自 steelman 反面**當 filter；貴的**異質模型 skeptic**（別家/別代、prompt 明確 refute 非 confirm）只給最大 call。
- ★落地：把 reviewer(對抗審)角色**指定跑不同模型/代 + refute prompt**，才有框外效果——同模型審 = 框內，救不了 groupthink。

## 5. 驗收模型：user-in-loop vs autonomous

- **mailbox in-loop（用戶盯著）**：砍正式 QA stage，**判斷角色持 release-pass 權**（沒問題過、有問題才升用戶）。但持**真 bar 非橡皮章**。
- **autonomous（fire N 條走開，如 LG 下游）**：正式 QA 硬閘回歸（走開時的安全網）。
- **量測角色產標準全探針數字**（全維度一次抓齊、結構化）→ 判斷角色判得動、不 bounce。maker/checker：量測產數字、判斷角色判。

---

這五條是這套工作流「為何有效」的內核。1-3 是框內紀律（個體角色守），4 是框外結構（判斷層需異質對手），5 是驗收分軌。**最該記住 4**：這套放大人的判斷，品質綁在「判斷層有無異質挑框」——同模型自我同意會自信地產出錯結論。
